AI coding agents write changes fast. sdlc is the layer that decides whether those changes are safe to ship — a terminal-native control plane that drives any change through a 25-gate Secure-SDLC pipeline, demands evidence at every gate, runs an independent cross-model red-team, and computes a deterministic release verdict. Models suggest; the orchestrator decides; evidence proves.

The problem

Speed without governance is unaudited risk.

AI agents (Claude Code, Codex, Copilot, Gemini, local models) generate and modify code at a pace humans can't review line-by-line. What's usually missing isn't capability — it's governance:

  • No enforced evidence that a change was scoped, threat-modeled, tested, scanned, and adversarially reviewed.
  • No tamper-evident record of who or what decided a change was safe.
  • Reports that assert “production-ready / secure” with nothing behind the claim.
  • A single model reviewing its own output — repeating its own blind spots.

For anything touching security, privacy, infrastructure, or regulated domains, that gap means shipping change you can't actually audit or attribute.

Impact

The gap is categorical, and growing.

AI-assisted coding is now mainstream across the industry, and the volume of machine-generated change keeps rising. Every team adopting these agents on consequential code inherits the same governance gap above. We deliberately don't cite a headline statistic here — the point is categorical, not a number we can defend. Claim discipline applies to our own marketing too.

How sdlc solves it

A control plane over your AI workers.

sdlc doesn't replace your coding agent — it governs it. Any of Claude Code / Codex / Gemini / Kimi / local Ollama / a custom CLI becomes a worker assigned to a role, behind gates the orchestrator owns.

25-gate pipeline

Intake → threat model → implementation → scans → red-team → attestation → deploy → final report. Each gate has an owner, required artifacts, and a verdict computed from evidence — not prose.

Tamper-evident ledger

Every decision is appended to an event log with chained SHA-256 digests. Tampering is detected in every validation mode.

Cross-model red-team

For high-risk work the reviewer must be a different model than the implementer — diversity catches what a monoculture repeats.

Claim discipline

Reports can't assert “production-ready / 100x / compliant” without linked evidence. The tool even gated its own development NO_GO.

Deterministic verdict

GO / NO_GO / GO_WITH_ACCEPTED_RESIDUAL_RISKS is computed from evidence. A CRITICAL/HIGH finding can never be “accepted” into a positive verdict — it must be fixed.

Signed, evidenced releases

Releases ship a CycloneDX SBOM and a Sigstore-keyless signature, verifiable with cosign.

Is there no alternative?

There are adjacent tools — none we found combine all of this.

Honest answer: yes, alternatives and adjacent tools exist, and sdlc uses several of them rather than replacing them:

  • CI/CD (GitHub Actions, GitLab CI) — runs steps, but has no risk-classified gate model or evidence verdict.
  • Policy-as-code (OPA / Conftest) — enforces rules, but isn't an SDLC orchestration layer over AI agents.
  • Scanners (Bandit, detect-secrets, pip-audit, Checkov) — sdlc orchestrates these as gate evidence.
  • Supply-chain provenance (SLSA, Sigstore) — sdlc adopts Sigstore for release signing.
  • The coding agents themselves — which produce changes but don't govern or adversarially review them.

What we have not found is a single open tool that combines a risk-classified 25-gate pipeline + a tamper-evident evidence ledger + enforced cross-model red-team + claim discipline + a deterministic release verdict, framed specifically as a control plane over AI coding agents. We do not claim none exists — only that this combination is uncommon.

Why use it

Evidence you can audit, claims you can trust.

Measured, not asserted

A built-in benchmark scores the tool on real dimensions and labels what it can't measure UNAVAILABLE. (Scores are corpus-relative, not an absolute grade.)

Multi-model by default

Assign the best model per role; force cross-model independence for high-risk work; fall back honestly when a worker is unavailable.

Local-first & offline

Planning, gates, the TUI, benchmark, and validation run with no network. Nothing leaves your machine by default.

Honest by construction

It refuses to overstate — including about itself. The committed comparative figure is a measured ~4× on a small (4-run) corpus, not “100×” (artifacts/bench/comparative.json).

What's unique

It governs AI agents — and holds itself to the same bar.

The distinctive thing isn't any single gate; it's the combination as a governance layer over AI coding agents, plus a culture of claim discipline the tool enforces on itself. During its own development it ran its own 25-gate pipeline, found a real CRITICAL in its own verdict logic, gated itself NO_GO, and only reported what an independent red-team could reproduce. A governance tool that rubber-stamped itself would be the failure mode.

How to use it

Ten minutes, no AI calls, no cost.

$ git clone https://github.com/gwuml/sdlc && cd sdlc
$ python3 -m venv .venv && . .venv/bin/activate && pip install -e .

$ python -m sdlc init
$ python -m sdlc plan "add OAuth login with audit logging" --risk auto
$ python -m sdlc status <run-id>     # 25 gates, local vs release state
$ python -m sdlc next <run-id>       # the safest next step toward GO
$ python -m sdlc tui <run-id>        # interactive dashboard (q to quit)

Then read, in order: README → USAGE.md (install + 25 worked transcripts) → WHY_THIS_TOOL → FEATURE_GATE_MAP. Install a signed release by verifying it first — see USAGE.md.

How to develop it further

Extend the workers, gates, and benchmark.

  • Add a worker family: implement an adapter in sdlc/adapters.py (invoked via subprocess with a sanitized environment, prompt over stdin) and register it. Custom CLIs are registered in policy. (execv + binary-hash pinning is planned for the Rust port, FAC-19 — not in the current Python path.)
  • Add or tune a gate: gate definitions live in sdlc/pipeline.py (the canonical 25); deterministic checks in sdlc/engine.py.
  • Add a benchmark dimension: sdlc/bench.py, tagged by kind so only real corpus measurements enter the headline.
  • Contribute: tests gate every PR (.github/workflows/ci.yml); see CONTRIBUTING.md, AGENTS.md (roles + hard safety rules), and SECURITY.md.

There's also a Rust foundation (src/) for an eventual single binary — today it's an early, parity-tested library skeleton (models + 25-gate pipeline + verdict), not a working CLI.

What it is NOT good for

Where sdlc is the wrong tool.

Not a coding agent

It won't write your code better than Claude Code or Codex. It governs them; it doesn't out-code them.

Not for throwaway work

For prototypes, scripts, and spikes, a 25-gate pipeline is overkill and will feel like friction.

Not a deploy bot

Production deploy is locked by default and requires explicit human approval + rollback evidence. It will not auto-ship.

Not a correctness guarantee

It evidences and gates; it does not prove your code is correct, secure, or compliant. Those claims need real evidence you supply.

Who should not use it (yet)

  • Solo developers on toy or personal projects who want zero friction.
  • Teams wanting fully-autonomous, unsupervised AI that ships without humans in the loop.
  • Windows-only shops — only macOS/Linux are exercised today.
  • Anyone needing a hardened, externally-audited, battle-tested product right now (see gaps below).

Security — reviewed vs. not

Be precise about what's been checked.

Reviewed done

  • An internal automated security review of the change set raised no high-confidence exploitable findings (internal, not an external pentest — see the right column).
  • Multiple independent, hostile red-team rounds culminating in a reproduced GO on clean clones (see docs/SESSION_AUDIT_PROMPT.md, PR #3).
  • Verdict integrity (a CRITICAL/HIGH can't be accepted into a positive verdict).
  • Secret redaction (PEM / PuTTY / SSH2 / JWK / tokens) with fuzz tests.
  • Tamper-evident ledger; HMAC actor-proof for finding closure; workers can't mutate the control-plane ledger.
  • Release-signature verification (cosign verify-blob → Verified OK on the real v0.2.0).

NOT reviewed / not validated open

  • No external or professional third-party penetration test.
  • Real worker execution against live LLMs end-to-end, and deploy execute against a real target.
  • Cross-platform (Windows) and the Rust binary path.
  • The security of the operator-installed worker CLIs themselves (Claude Code, Codex, …) — outside this tool's boundary.
  • Performance / resource behavior on large repositories.

Report vulnerabilities privately per SECURITY.md — not via public issues.

Honest status: sdlc is a young, open-source tool (v0.2.0, Apache-2.0). It is adoptable and trustworthy to install — licensed, CI-gated, documented, signed, and independently red-teamed — but it is not a hardened, externally-audited, production-proven product, and it has not yet been validated doing its core job end-to-end with live AI workers. Every number on this page is measured and intentionally un-inflated; where we couldn't measure something, we said so.
